## T-AUTH — Аутентификация

### T-AUTH-01: Первый запуск без ключа
```
Условие:  ~/.config/gemini-chat/history.db не существует,
          в Keychain нет записи "gemini-chat/gemini"
Действие: Запустить приложение
Ожидание: Автоматически открывается диалог настроек
          Кнопка "Отправить" задизейблена
          Статус: "⚠️ API ключ не задан"
```

### T-AUTH-02: Неверный API ключ
```
Условие:  Пользователь вводит в поле ключа "invalid-key-123"
Действие: Подождать 500ms после ввода (дебаунс)
Ожидание: "❌ Ключ недействителен: 401 Unauthorized"
          Список моделей остаётся задизейблен
          Кнопка "Сохранить" — НЕ сохраняет невалидный ключ
```

### T-AUTH-03: Валидный ключ — загрузка моделей
```
Условие:  Пользователь вводит корректный API ключ (≥ 39 символов)
Действие: Подождать 500ms дебаунс + время запроса
Ожидание: "✅ Ключ рабочий, доступно моделей: N"
          Выпадающий список заполнен реальными именами моделей
          Предвыбрана "Gemini 2.0 Flash" (или первая доступная)
```

### T-AUTH-04: Повторный запуск с сохранённым ключом
```
Условие:  Ключ сохранён в Keychain с прошлого сеанса
Действие: Запустить приложение
Ожидание: Диалог настроек НЕ открывается
          В фоне тихо выполняется ValidateKey()
          Через 1-3 сек статус: "✅ Готов: gemini-2.0-flash"
          Если ключ всё ещё валиден — приложение сразу готово к работе
```

### T-AUTH-05: Устаревший ключ (отозван пользователем в Google AI Studio)
```
Условие:  Ключ сохранён в Keychain, но отозван в Google AI Studio
Действие: Запустить приложение → тихая валидация → 401
Ожидание: Баннер "⚠️ API ключ недействителен"
          Кнопка [Настройки →] в баннере
          Отправка сообщений заблокирована до обновления ключа
```

### T-AUTH-06: Удаление ключа
```
Условие:  Ключ сохранён, открыты настройки
Действие: Нажать "🗑 Удалить ключ" → подтвердить в диалоге
Ожидание: Ключ удалён из Keychain
          Поле ключа очищается
          Статус меняется на "⚠️ Ключ не задан"
          Проверить: keyring.Get("gemini") → возвращает ErrNotFound
```

## T-NET — Сетевые ошибки и лимиты

### T-NET-01: Нет интернета при отправке
```
Условие:  Отключить Wi-Fi / Ethernet
Действие: Написать сообщение и нажать "Отправить"
Ожидание: Через ~10 секунд (timeout) появляется ошибка
          "❌ Нет подключения к сети"
          Кнопка "Отправить" снова активна
          Незавершённое сообщение пользователя — НЕ сохраняется в БД
          (или помечается как "не отправлено")
```

### T-NET-02: Нет интернета при запуске (ключ уже сохранён)
```
Условие:  Отключить сеть, запустить приложение
Действие: Запуск
Ожидание: ValidateKey() завершается по timeout
          Баннер: "⚠️ Не удалось проверить ключ — нет сети"
          История чатов из SQLite загружается и доступна для просмотра
          Отправка сообщений недоступна (понятная причина)
```

### T-NET-03: Rate limit (429 Too Many Requests)
```
Условие:  Бесплатный тир — 15 RPM исчерпан
Действие: Отправить сообщение при исчерпанном лимите
Ожидание: "⏳ Превышен лимит запросов. Подождите 60 секунд."
          Опционально: обратный отсчёт в статусной строке
          Кнопка "Отправить" блокируется на время ожидания
```

### T-NET-04: Нестабильный интернет (обрыв во время стриминга)
```
Условие:  Начать получение стримингового ответа, затем обрыв сети
Действие: Отправить длинный запрос → на середине ответа отключить сеть
Ожидание: Частичный ответ отображён
          Ошибка "❌ Соединение прервано"
          Частичный ответ НЕ сохраняется в БД
          (или сохраняется с пометкой "[ответ обрезан]")
          Кнопка "Отправить" снова активна
```

### T-NET-05: Timeout (сервер не отвечает)
```
Условие:  Gemini API недоступен (503 Service Unavailable)
Действие: Отправить запрос
Ожидание: Через 30 секунд (configurable timeout): "❌ Сервер не отвечает (503)"
          Предложение повторить: кнопка [Повторить]
```

## T-STREAM — Потоковый вывод

### T-STREAM-01: Базовый стриминг
```
Условие:  Нормальная работа
Действие: Отправить "Напиши список из 10 языков программирования"
Ожидание: Текст появляется постепенно, слово за словом / предложение за предложением
          Скролл следует за текстом автоматически
          Кнопка "⏹ Стоп" видна во время стриминга
```

### T-STREAM-02: Отмена стриминга
```
Условие:  Идёт получение длинного ответа
Действие: Нажать кнопку "⏹ Стоп"
Ожидание: Стриминг прекращается через < 500ms
          Частичный ответ остаётся видимым
          Кнопка меняется обратно на "▶ Отправить"
          ctx.Cancel() вызван → горутина завершается
          Частичный ответ НЕ сохраняется в БД
```

### T-STREAM-03: Markdown во время стриминга
```
Условие:  Нормальная работа
Действие: Спросить "Покажи пример кода на Go с объяснением"
Ожидание: Пока стримится — обычный текст может выглядеть "сырым"
          После получения блока ```go ... ``` — рендерится как код
          По завершении — весь MD отформатирован корректно
          Незакрытые токены (** без закрывающего **) не ломают UI
```

### T-STREAM-04: Одновременный стриминг (защита)
```
Условие:  Идёт получение ответа
Действие: Попытаться нажать "Отправить" снова (например, двойной клик)
Ожидание: Кнопка задизейблена → второй запрос невозможен
          Один чат — один активный запрос одновременно
```

## T-HIST — История чатов

### T-HIST-01: Сохранение после перезапуска
```
Условие:  Провели несколько диалогов
Действие: Закрыть приложение → открыть снова
Ожидание: Список чатов восстановлен в том же порядке
          Открыть любой чат → все сообщения на месте
          Заголовки чатов соответствуют первым сообщениям
```

### T-HIST-02: Контекст истории в новом запросе
```
Условие:  Чат с 5 сообщениями: "Меня зовут Алексей"
Действие: Написать "Как меня зовут?"
Ожидание: Gemini отвечает "Алексей" (помнит контекст)
          Вся история передаётся в API как contents[]
```

### T-HIST-03: Переключение между чатами
```
Условие:  Два чата с разными темами
Действие: Быстро переключаться между чатами 5 раз
Ожидание: Каждый чат показывает свою историю
          Нет смешивания сообщений между чатами
          Нет утечек памяти (VBox очищается при переключении)
```

### T-HIST-04: Удаление чата
```
Условие:  Чат с 20 сообщениями
Действие: Нажать "🗑 Удалить" → подтвердить
Ожидание: Чат исчезает из sidebar
          В SQLite: строки из chats и messages удалены (CASCADE)
          Если удалён активный чат → открывается первый оставшийся
          Если чатов не осталось → пустое состояние "Создайте новый чат"
```

### T-HIST-05: Автозаголовок
```
Условие:  Создать новый чат
Действие: Отправить "Объясни разницу между slice и array в Go"
Ожидание: Заголовок чата в sidebar: "Объясни разницу между slice и..."
          Длина: ≤ 45 символов + "..."
```

## T-UI — Пользовательский интерфейс

### T-UI-01: Пустое сообщение
```
Условие:  Поле ввода пустое или содержит только пробелы
Действие: Нажать Enter или кнопку "Отправить"
Ожидание: Запрос НЕ отправляется
          Никаких ошибок не показывается
          Поле остаётся в фокусе
```

### T-UI-02: Очень длинное сообщение пользователя
```
Условие:  Нормальная работа
Действие: Вставить текст > 5000 символов и отправить
Ожидание: Сообщение отображается в пузырьке с переносом (TextWrapWord)
          Пузырёк не выходит за пределы окна
          Скролл работает корректно
```

### T-UI-03: Смена темы
```
Условие:  Открыть настройки
Действие: Переключить с "Системная" на "Тёмная"
Ожидание: Тема применяется немедленно без перезапуска
          Все виджеты обновляются (sidebar, пузырьки, toolbar)
          При следующем запуске тема восстанавливается
```

### T-UI-04: Изменение размера окна
```
Условие:  Нормальная работа
Действие: Уменьшить окно до 600×400, потом увеличить до полного экрана
Ожидание: Текст переносится корректно на любом размере
          Sidebar и chatview масштабируются пропорционально
          Поле ввода всегда видно внизу
```

## T-PERF — Производительность

### T-PERF-01: Длинный контекст (много сообщений в одном чате)
```
Условие:  Чат с 200 сообщениями
Действие: Открыть чат
Ожидание: Загрузка < 1 секунды
          Прокрутка плавная (≥ 30 FPS)
          Память: < 200 МБ
Реализация: Загружать последние 50 сообщений, остальные по запросу (пагинация)
```

### T-PERF-02: Долгий стриминг (большой ответ)
```
Условие:  Попросить написать 2000 слов эссе
Действие: Отправить запрос
Ожидание: UI не замерзает весь период стриминга (60-90 секунд)
          ParseMarkdown(accumulated) вызывается не чаще 1 раза / 50ms
          CPU: < 30% на M1/M2 MacBook
```

### T-PERF-03: Нагрузка SQLite при частых сообщениях
```
Условие:  Отправить 50 сообщений подряд быстро
Действие: Спамить короткие запросы
Ожидание: Нет ошибок SQLITE_BUSY (busy_timeout = 5000ms)
          Все сообщения сохранены корректно
          Нет дублирования записей
```

### T-PERF-04: Память при большом количестве чатов
```
Условие:  1000 чатов в БД
Действие: Запустить приложение
Ожидание: Sidebar загружает только метаданные (без messages)
          Память при старте: < 100 МБ
          ListChats() выполняется за < 100ms
```

### T-PERF-05: CPU при простое
```
Условие:  Приложение открыто, ничего не происходит
Действие: Оставить на 5 минут
Ожидание: CPU: < 1% (Fyne не делает постоянных перерисовок в idle)
```

## T-CROSS — Кросс-платформенность

### T-CROSS-01: macOS Apple Silicon (arm64)
```
Действие: go build / fyne package на M1/M2/M3
Ожидание: Бинарник собирается без ошибок
          GeminiChat.app запускается
          Keychain работает
          SQLite работает (modernc.org/sqlite поддерживает darwin/arm64)
```

### T-CROSS-02: macOS Intel (amd64)
```
Действие: GitHub Actions job: build-macos на macos-13 runner (Intel)
Ожидание: Аналогично T-CROSS-01
```

### T-CROSS-03: Windows 10/11 (amd64)
```
Действие: GitHub Actions job: build-windows
Ожидание: GeminiChat.exe собирается
          Windows Credential Manager используется для ключей
          SQLite работает
          Нет иконки командной строки (windowsgui в ldflags или fyne package)
```

### T-CROSS-04: Ubuntu 22.04 LTS (amd64)
```
Действие: GitHub Actions job: build-linux
Ожидание: Бинарник собирается (нужны libgl1-mesa-dev, xorg-dev)
          Запускается в GNOME (X11 и Wayland через XWayland)
          SQLite работает
```
